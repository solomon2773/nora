# Nora README & Proof Asset Plan

Goal: make the public repo and site feel unmistakably open source, self-hostable, operator-real, and honest about runtime direction.

## Messaging rules

1. Lead with **open source** and **self-hosting**
2. Make it explicit that **commercial use by anyone is allowed under Apache 2.0**
3. Describe OpenClaw as the **strongest supported runtime today**, not the permanent only-runtime story
4. Add more real operator screenshots, especially the agent-facing and onboarding surfaces
5. Keep the repo as the trust anchor; use the live app as supporting proof

## Priority proof assets

### 1. README screenshots
Show:
- landing page with OSS + self-host framing
- pricing/support page with Apache 2.0, commercial-use, and managed-path language
- signup or onboarding page proving operator flow exists
- later: agent list, deploy flow, settings, and agent detail surfaces

### 2. Runtime direction proof
Show:
- OpenClaw as the concrete runtime example today
- wording that keeps future runtime integrations plausible
- terminology that avoids permanent single-runtime lock-in

### 3. Self-host proof
Show:
- quick start
- raw GitHub install scripts
- Docker Compose path
- operator account flow

## Priority copy

- **Landing:** Nora is the open-source control plane for agent operations.
- **Pricing/support page:** Apache 2.0, self-hostable, commercially usable by anyone, with paid support and managed-path context.
- **README:** fully open source, self-hostable, commercially usable by anyone, OpenClaw-first today, broader runtime direction over time.
- **Runtime direction:** strongest with OpenClaw today, not permanently OpenClaw-only.

## What to avoid

- maintainer-commercial framing as the repo's main story
- enterprise packaging language as the front door
- unsupported claims
- implying Nora can only ever work with OpenClaw
- screenshot plans that stay purely marketing-facing and never prove operator reality
